# AI主体クイズ生成ワークフロー

AIが主体となってクイズのテーマ決定・調査・作成・品質担保を行い、最後に人のチェックを受けてpushする。
ファイル形式・テンプレート・タグ・スタイルの詳細ルールは `contents/quiz/AGENTS.md` に従うこと（このskillでは繰り返さない）。

## ワークフロー全体

```
進行チェックリスト:
- [ ] Step 1: 対象日付と曜日テーマの確定
- [ ] Step 2: テーマ候補の提案と壁打ち（ユーザーと対話）
- [ ] Step 3: 公式情報の調査・ファクトチェック
- [ ] Step 4: クイズファイルの作成
- [ ] Step 5: 品質検証（動作検証・lint・typecheck・URL確認）
- [ ] Step 6: 人のレビュー（開発サーバー起動＋承認を得るまでpushしない）
- [ ] Step 7: commit & push
```

## Step 1: 対象日付と曜日テーマの確定

1. **対象日付の決定**:
   - ユーザーが日付や期間を指定した場合はそれに従う（例:「1/20の分」「来週1週間分」「明日から3日分」）。
   - 指定がなければ当日の日付を対象とする。
2. 対象日付にすでにクイズファイルが存在しないか確認する。存在する場合はユーザーに新規作成か更新かを確認する:

```bash
ls contents/quiz/YYYY/MM/DD.tsx 2>/dev/null
```

3. 対象日付の曜日を確認し（macOS: `date -j -f "%Y-%m-%d" "YYYY-MM-DD" +%A` / Linux: `date -d "YYYY-MM-DD" +%A`）、曜日テーマを決定する（曜日タグの一覧は `contents/quiz/AGENTS.md` の Tag Policy を参照）。
4. **複数日分をまとめて作る場合**（例: 1週間分の先行作成）:
   - 対象日付をすべて列挙し、それぞれの曜日テーマを一覧にしてユーザーに提示してから着手する。
   - テーマ選定の壁打ち（Step 2）は全日分まとめて行ってよいが、Step 3以降の調査・作成・検証は1問ずつ丁寧に行う。
   - レビュー（Step 6）は全問完成後にまとめて依頼してよい。
   - 将来日付のクイズは `created_at` がその日付になるため、pushしてもサイト上は公開日まで表示されない（先行作成しても問題ない）。

## Step 2: テーマ候補の提案と壁打ち

AIが主体的にテーマ候補を出し、ユーザーと壁打ちして決める。

1. **重複チェック**: 対象曜日タグの直近のクイズ（10〜20問程度）のタイトルを確認し、最近扱ったトピックを把握する:

```bash
rg -m 1 -o 'title: "([^"]+)"' -r '$1' contents/quiz/ -g '*.tsx' --sort path | tail -30
```

2. **候補提案**: 曜日テーマに沿ったテーマ候補を3〜5個提案する。各候補には以下を添える:
   - テーマ名と扱う論点（1〜2文）
   - 想定難易度（基礎/中級/応用）
   - 選定理由（最近の技術動向、既存クイズとの関連・未カバー領域など）
3. AskQuestionツールで候補を提示してユーザーに選んでもらう。ユーザーが別案や方向修正を出したら、それを踏まえて再提案し、合意できるまで対話する。
4. テーマ確定後、問題の切り口（何を問うか、どんな誤答を混ぜるか）を1〜2文で確認してからStep 3に進む。

## Step 3: 公式情報の調査・ファクトチェック

出題内容の正確性を担保するため、作成前に必ず一次情報を確認する。

- 公式ドキュメントを直接読む（WebFetch / Web検索を使用）。優先順位: 公式ドキュメント > 公式ブログ > 信頼できる技術資料。
- AWS関連のテーマは AWS Knowledge MCP（`aws___search_documentation` / `aws___read_documentation`）を活用する。
- 確認すべきこと:
  - 出題する仕様・機能・制限値が現時点の公式情報と一致しているか（エディション差、リージョン差、非推奨化に注意）
  - 誤答選択肢が「もっともらしいが確実に誤り」であること（曖昧な選択肢は作らない）
  - referencesに使うURLが実在しアクセス可能であること
- 調査結果は要点をユーザーに共有してから作成に進む（認識ズレの早期発見のため）。

## Step 4: クイズファイルの作成

- `contents/quiz/YYYY/MM/DD.tsx` に作成する。テンプレート・タグ・スタイル・解説の書き方はすべて `contents/quiz/AGENTS.md` に従う。
- **author（AI生成時）**: AIが主体で作成したクイズは、人が単独で書いたものと区別できるよう `jimatomo with <モデル名>` 形式にする（例: `jimatomo with Composer`）。`author_url` は従来どおり `https://github.com/jimatomo` でよい。使用中のモデル名が分からない場合はユーザーに確認する。
- `created_at` / `updated_at` は対象日付（Step 1で確定した日付）を設定する。
- 新規タグが必要な場合のみ `app/quiz/lib/tags.ts` に追加する（同一コミットに含める）。

### 解説の教育価値（one quiz, one takeaway）

- 解説を正答・誤答の言い換えだけで終わらせず、読者が持ち帰れる学びを最低1つ入れる。
- 学びはテーマに応じて、**実務での判断軸、短い具体例、よくある落とし穴、類似機能との使い分け**のいずれかから最も有用な1点を選ぶ。
- 網羅的な説明を詰め込まない。標準では「根拠の説明＋持ち帰り1点」に絞り、短時間で読み切れる分量にする。追加の背景が問題を理解するために必要な場合だけ詳しくする。
- 仕様の羅列より「この知識をいつ、どう使うか」が伝わる説明を優先する。例えば、単に機能差を列挙するのではなく、どの条件ならどちらを選ぶかを1つ示す。

## Step 5: 品質検証

作成後、以下をすべて実施する。失敗したら修正して再実行し、全部通るまでStep 6に進まない。

1. **動作検証（該当する場合は必須）**: クイズにSQL・コード・コマンドの実行結果を問う内容が含まれる場合、ローカルで実際に実行して正解を裏取りする。
   - SQL → DuckDB CLI（`duckdb -c "..."`）で検証。Snowflake固有構文でDuckDBで再現できない場合は、公式ドキュメントの記載と照合したことを明示する
   - Python → `python3 -c "..."` やスクリプトで検証
   - シェル/Linuxコマンド → 安全なものはローカルで実行して確認
2. **静的チェック**:

```bash
pnpm run lint && pnpm run typecheck
```

3. **URL確認**: references の全URLに実際にアクセスし、リンク切れ・リダイレクト先の内容違いがないか確認する。
4. **セルフレビュー**: `contents/quiz/AGENTS.md` の Quality Checklist を一通り確認する（特に: 曜日タグと日付の一致、選択数を最小にする出題形式、JSX内のアポストロフィは `&apos;`）。加えて、解説が次を満たすか確認する:
   - 正答・誤答の言い換えだけで終わっていない
   - 実務の判断軸・具体例・落とし穴・使い分けのいずれか1つが残る
   - 学びを増やしすぎず、主なtakeawayを1つに絞っている

## Step 6: 人のレビュー

**ユーザーの明示的な承認を得るまで、絶対にcommit・pushしない。**

レビュー依頼時は、ユーザーがブラウザで実機確認しやすいよう **開発サーバーを起動してから** 依頼する。

1. **開発サーバーの起動**（未起動の場合）:

```bash
pnpm run dev
```

   - 既に起動中なら再利用し、二重起動しない。
   - 起動後、対象クイズのURLをレビュー依頼に含める（例: `http://localhost:3000/quiz/2026/07/11`）。
   - 将来日付のクイズは一覧から隠れる場合があるが、日付パス直打ちでは確認できる。確認できない場合はその旨を伝える。

2. **レビュー依頼時にまとめて提示する内容**:

- 対象クイズのローカルURL（上記）
- 問題文・選択肢・正解・解説の要約（ファイルを開かなくても判断できる粒度）
- 正解の根拠（参照した公式情報の該当箇所）
- 実施した動作検証の内容と結果
- lint / typecheck の結果

修正指摘を受けたら反映し、影響範囲に応じてStep 5の検証を再実行してから再度レビューを依頼する。

## Step 7: commit & push

承認を得たら、mainブランチへ直接commit & pushする。

```bash
git add contents/quiz/YYYY/MM/DD.tsx  # tags.ts を更新した場合はそれも含める
git commit -m "content: <クイズタイトル>に関する問題を追加"
git push origin main
```

複数日分をまとめて作成した場合は、対象のクイズファイルをすべてaddし、1コミットにまとめてよい（例: `content: 1/20〜1/26の1週間分の問題を追加`）。

push完了後、対象日付・タイトル・コミットハッシュを報告する。
