# 継続改善ループ

レビュー後のフィードバックを、次回のクイズ生成品質へつなげるための手順。目的はログを増やすことではなく、手戻りを減らしながら品質を上げることにある。

## ループ

`Observe → Record → Decide → Change → Verify → Re-observe` の順で回す。

1. **Observe**: 人の指摘、レビュー往復、検証失敗、AIが気づいた迷いを観測する。
2. **Record**: 再利用可能な事実だけを当年の `feedback/YYYY.md` に追記する。
3. **Decide**: 証拠と重大度から、候補のまま残すか標準手順へ昇格するかを決める。
4. **Change**: 合意した最小の変更を、最も近い1箇所へ反映する。
5. **Verify**: 構文・リンク・矛盾・実行可能性を確認する。
6. **Re-observe**: 次回の同種タスクで成功指標を確認し、採用・再設計・撤回を判断する。

## 記録対象

次のいずれかに該当するものだけを記録する。

- レビューで修正が発生し、別のクイズでも再発しそうなもの
- 調査、作成、検証、レビューの手戻りを生んだもの
- 正確性、安全性、公開可否に影響する見落とし
- ユーザーが今後も維持したいと明示した進め方
- 既存の手順が役立った、または役立たなかったことを示す観測結果

表現上の単発の好み、会話全文、個人情報・秘密情報、根拠のない推測は記録しない。

## ログの書式

当年のファイル `feedback/YYYY.md` に次の単位で追記する。IDは `YYYY-MM-DD-連番` とする。

```markdown
## 2026-07-12-01: 短い題名

- Quiz: YYYY-MM-DD / クイズタイトル
- Source: user-feedback | review-observation | validation
- Phase: theme | research | authoring | validation | review | publish
- Signal: 起きた事実と影響
- Cause: 根本原因。未確定なら「仮説」と明記
- Action: 今回の対処、または次に試す最小変更
- Success metric: 次回、改善したと判断できる観測可能な条件
- Tags: `短い分類`, `再利用しやすい分類`
- Status: candidate | proposed | adopted | rejected
- Related: 関連ID。なければ none
```

1件を短く保ち、同じ原因の追記は新規ルールを増やさず `Related` で結ぶ。年が変わったら同じ書式で新しい年ファイルを作る。

## 昇格基準

次のいずれかを満たす場合に `proposed` とし、ユーザーへ変更案を示す。

- 同じ原因の問題が異なるクイズで2回以上起きた
- 正確性、安全性、機密性、誤公開に関わる重大な問題である
- ユーザーが恒常ルール化を明示した
- 変更コストが小さく、既存の手順と矛盾せず、効果を観測できる

それ以外は `candidate` のまま残す。候補数を増やすことを成果にしない。既存ルールで防げた問題は新しいルールを足さず、既存ルールの配置・明瞭さ・実行タイミングを直す。

## 変更先の選び方

- `SKILL.md`: 全クイズに共通する順序、停止条件、意思決定
- `references/`: 詳細な評価基準、長い例、蓄積データ
- `contents/quiz/AGENTS.md`: TSX形式、タグ、文体などクイズ成果物の恒常ルール
- `scripts/`: 同じ機械的検査を繰り返し、手作業では漏れやすい場合

詳細を `SKILL.md` へ詰め込みすぎない。同じ規則を複数ファイルに複製せず、他の場所からは参照する。

## 変更と検証

1. 変更前に対象ID、根拠、期待する成功指標を提示し、ユーザーの合意を得る。
2. 1回の改善では原則1つの原因だけを扱い、差分を小さくする。
3. 変更後に以下を確認する。
   - YAML frontmatterとMarkdownリンクが壊れていない
   - `SKILL.md` のチェックリスト、見出し番号、停止条件が矛盾していない
   - 新しい指示が実行可能で、既存指示と重複していない
   - `git diff --check` が通る
4. skill-creatorの `quick_validate.py` が利用できる場合は、スキルディレクトリに対して実行する。
5. ログを `adopted` に更新し、変更先を `Action` に記す。採用しない場合は理由を添えて `rejected` にする。

## 次回の効果確認

次回のStep 7で、`adopted` かつ未確認の項目を先に探し、成功指標を評価する。

- 成功: 効果をログへ追記し、採用を維持する。
- 不明: 追加観測が必要な理由を追記し、判断を保留する。
- 失敗または副作用あり: 変更を撤回または再設計し、新しい関連IDで結果を残す。

古くなった候補は無理にルール化しない。年次ファイルが大きくなった場合は、直近20件と関連タグだけを読み、必要なIDを追加で参照する。
